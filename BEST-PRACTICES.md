# Best Practices — Task Admin

## Универсальный режим (важно)

- Приоритет у ТЗ пользователя и фактической структуры репозитория. Не фиксируйся на одном стеке по умолчанию.
- Если в ТЗ указаны React/Next/Vue/Nuxt/Nest/FastAPI/Django/Rails/Go или другой стек, реализуй и настраивай задачу в этом стеке.
- Если в проекте не хватает базовой конфигурации (типы, стили, линтер, роутинг, провайдеры, зависимости) — добавляй setup-шаги и команды установки.
- По возможности держи новые файлы компактными (ориентир до 100 строк), но не ломай архитектуру ради этого.
- Для сложных AI-сценариев строй слой как `planner -> artifact graph -> workflow runtime`, а не как набор keyword-эвристик.
- Если пользователь ссылается на прошлые результаты (`это изображение`, `добавь туда`, `используй прошлый файл`), сначала опирайся на реальные недавние артефакты и последние успешные workflow run.
- Если пользователь просит улучшить уже созданный HTML/PPTX/документ, редактируй текущий артефакт как продолжение, а не создавай новый бренд, новый стиль и новый файл без прямой команды.
- Для UI и лендингов избегай generic hero-only заготовок: сразу строй внятную композицию, контраст, типографику, секции с разной плотностью и аккуратный responsive layout.
- Если в задаче несколько изображений или медиа-ассетов, используй их все явно и по-разному; не дублируй один и тот же визуал под разными именами и не оставляй пустые медиа-слоты.
- Если уточнение всё же нужно, задавай один узкий вопрос из `scenario plan + artifact graph + workflow memory`; не превращай ответ в общую анкету про аудиторию, стиль и дедлайн без причины.

## Native MCP и интеграции

- При добавлении native-интеграции сразу закладывай 4 слоя: канонические методы, alias-карту по пользовательскому языку, chat-hydration актуальных tools из кода и formatter реального RESULT.
- Для чата MCP-интеграции должны попадать в LLM как реальные function/tool definitions, а не только как текст в prompt. Prompt помогает выбрать инструмент, но не заменяет структурный tool-calling.
- Это правило общее для всех провайдеров чата и всех MCP-серверов: OpenAI/Codex, OpenRouter и другие совместимые маршруты должны получать один и тот же актуальный список enabled tools.
- Имена tools, которые не проходят в provider как есть, кодируй/декодируй обратимо в одном месте. Нельзя дублировать разные схемы имён в prompt, runtime и UI.
- Alias-карта обязательна там, где пользователь и API говорят по-разному: `deals -> leads`, `boards -> databases`, `folders -> list_files`, `send_file -> sendDocument`.
- Делай semantic aliases для каждой публичной операции интеграции, если пользователь почти наверняка будет формулировать её иначе, чем канонический API-метод.
- В `call_tool` всегда своди alias к canonical method в одном месте. Не размазывай это по prompt, runtime и UI отдельно.
- В hydration и access control alias должен наследовать `enabled/access_mode` от канонического метода. Иначе alias обходит настройки коннектора.
- Read-only методы по умолчанию должны быть `enabled`, а write/delete/send — `ask`, если продукт явно не требует другого. Не заставляй модель или пользователя подтверждать обычное чтение данных.
- Метаданные и справочники описывай жёстко: `pipelines/statuses/users/databases/chats` нельзя подменять ими списки сделок, задач, файлов или отправку.
- Для внешних систем нельзя выдумывать строки таблиц, статусы, ID, файлы, чаты, ссылки и факт отправки. Либо есть реальный `MCP_CALL_JSON` и RESULT, либо одно узкое уточнение пользователю.
- Backend не должен маскировать неудачный tool-call доменными эвристиками вроде «уточни сделки или лиды». Если реального вызова не было, возвращай честную остановку или один фактологический вопрос без подмены намерения.
- Контекст tools в чате держи компактным: server key, имена, краткие описания и access mode. Не зашумляй prompt полными schema-дампами, если schema уже передана как function/tool definition.
- Если native tools меняются в коде, чат должен видеть свежий набор методов без ручного sync. Не опирайся только на устаревший сохранённый список tools.

## Лучшие практики по стеку (Rust, БД, Svelte)

### Общие принципы

1. **Одна итерация = готовый результат.** Не заглушки, не TODO вместо кода, не «сделаю потом». Полный рабочий слой: схема → миграция → сиды (если нужны) → API → фронт → навигация.
2. **Сразу подключать.** Новый файл/компонент/роут/API — сразу импорт, ссылка в навигации (+layout.svelte), вызов с фронта. Без подключения пользователь не увидит результат.
3. **Минимальный diff.** Меняй только то, что нужно для задачи. Точечные правки, без переписывания целых файлов и массового рефактора.

### Rust (backend, API)

- **Роуты:** все защищённые API — в группе с middleware (auth). Ответы: `Json(data)` при успехе, `api_error(StatusCode::*, "message")` при ошибке.
- **Ошибки:** не паниковать, возвращать осмысленные HTTP-коды (400, 404, 500) и текст ошибки в JSON.
- **Зависимости:** добавлять в `Cargo.toml` в нужную секцию `[dependencies]` или `[dev-dependencies]`. После изменения — сборка должна проходить (`cargo build`).
- **Парсинг/сериализация:** использовать `serde` (Serialize, Deserialize) для JSON; для форм/query — типизированные структуры.
- **Логирование:** предпочтительно `tracing` или `log`; не засорять stdout лишним выводом в production-пути.

### База данных (Drizzle, PostgreSQL, миграции, сиды)

- **Схема (Drizzle):** таблицы в `src/lib/db/schema.ts` (или аналог). Типы колонок — из `drizzle-orm/pg-core`. Связи через `relations()`.
- **Миграции:**
  - Файлы миграций — в каталоге `drizzle/` или `migrations/` (как в проекте). Имена — понятные, с датой/версией.
  - SQL должен быть идемпотентным где возможно: `ADD COLUMN IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`. Синтаксис под целевую СУБД (PostgreSQL).
  - При верификации выполняется `npm run db:migrate`. Ошибка миграции уходит в контекст повтора — исправь SQL и не ломай существующие таблицы.
- **Сиды (seed):**
  - Скрипт сидов — по конвенции проекта: `db:seed`, `seed` или `db:seeds` в `package.json`. Запускается после миграций при верификации.
  - Сиды должны быть идемпотентными (проверка существования, upsert), не падать при повторном запуске. При ошибке сидов задача повторяется с контекстом ошибки.
- **API и БД:** эндпоинты, которые читают/пишут БД, должны использовать одну и ту же схему и пул подключений (например, через `db` из `src/lib/db`). Не создавать отдельные подключения в каждом handler.
- **Переменные окружения:** `DATABASE_URL` для миграций и приложения. В скриптах верификации используется `DATABASE_URL` из окружения. Дефолт (если не задан): `postgresql://demo:demo@localhost:5432/demo` — совпадает с `demo/docker-compose.db.yml` (POSTGRES_USER/POSTGRES_PASSWORD/POSTGRES_DB).

### Запуск окружения (БД, бэкенд, фронт)

- **«Из коробки»:** при открытии проекта из админки и запуске **раннера** или **dev-сервера** (превью) окружение поднимается автоматически:
  - **Если в проекте есть `docker-compose.microcluster.yml`** — поднимается полный кластер в Docker (БД + фронт + gateway) и **локальный туннель** (по умолчанию localtunnel, *.loca.lt). Превью доступно по **внешнему URL**, который отображается в админке и сохраняется в профиле проекта.
  - Иначе, если есть `docker-compose.db.yml` (или `.yaml`) — выполняется `docker compose up -d`, ожидание готовности БД, затем при наличии в `package.json` скриптов `db:migrate` и `db:seed` — они выполняются. Фронт/бэк запускаются на хосте.
  Ручной запуск БД/миграций/сидов не требуется.
- **Один продукт = один кластер.** Каждый проект из админки (новый продукт) имеет свой микрокластер: своя БД, свой бэкенд, свой фронт, свой gateway. При «Запустить превью» поднимается только кластер этого проекта; туннель ведёт к его gateway.
- **Единый конфиг микрокластера:** `scripts/ensure-microcluster-config.sh` — единственный источник правды. Подставляет в каталог проекта актуальные `docker-compose.microcluster.yml` и `docker/nginx/microcluster.conf`. Вызывается автоматически: при scaffold (новый проект) и при каждом «Запустить превью» в админке. Ручное обновление проектов не нужно.
- **Scaffold** (`scripts/scaffold-sveltekit-cargo.sh`) создаёт каркас и вызывает `ensure-microcluster-config.sh`; также создаёт `scripts/tunnel.sh` и npm-скрипты `cluster:up`, `cluster:down`, `cluster:tunnel`. Туннель при «Запустить превью» поднимается автоматически (localtunnel, *.loca.lt).
- **Demo-проект:** в `demo/` есть `docker-compose.microcluster.yml` (БД + SvelteKit-фронт + nginx gateway). При «Запустить превью» поднимаются кластер и туннель; превью доступно по внешнему URL.
- **Бэкенд (Task Admin):** из корня репозитория `cargo run`.
- **Полный кластер (Docker):** в корне `docker compose -f docker-compose.microcluster.yml up -d` — БД (taskadmin), бэкенд (Rust), фронт (admin-ui), gateway. Другие учётные данные БД: `taskadmin`/`taskadmin`.
- Если БД не поднята, скрипты верификации пропускают миграции и проверяют только сборку.

### Svelte и SvelteKit

- **Структура:** страницы — `src/routes/+page.svelte`, layout — `src/routes/+layout.svelte`, API — `src/routes/api/.../+server.ts`. Общие компоненты и утилиты — `src/lib/`.
- **Роуты:** новый раздел приложения = новый путь в `src/routes/` (например, `src/routes/clients/+page.svelte`). Обязательно добавить ссылку в навигацию в `+layout.svelte`, иначе пользователь не попадёт на страницу.
- **Данные:** для данных с сервера — `+page.server.ts` и `load` с `fetch` или вызовом БД; для клиентского API — вызов из `+page.svelte` в эндпоинты (например, `/api/...`).
- **Стили:** использовать CSS-переменные проекта (например, `var(--surface)`, `var(--border)`). Не хардкодить цвета/шрифты из другого дизайн-система, если в проекте задана своя палитра.
- **Вёрстка и дизайн (сразу профессионально):**
  - **Сетка:** карточки, списки, блоки — сразу на CSS Grid. Контейнер: `display: grid`, `grid-template-columns` с repeat и minmax (например `repeat(auto-fill, minmax(280px, 1fr))` или явные брейкпоинты `1fr` / `repeat(2, 1fr)` / `repeat(3, 1fr)`), `gap` для отступов между ячейками. Не верстать колонки через float или только flex без сетки, если нужна ровная сетка карточек.
  - **Цвета и шрифты:** передовые палитры (нейтральные + один акцент, контраст текста по WCAG). Шрифты — системный стек плюс один подключаемый (Google Fonts: например Inter, Source Sans 3, Manrope, DM Sans) для заголовков и/или тела. Задавать через CSS-переменные (`--font-sans`, `--color-primary`, `--color-surface`) в :root или в существующем theme.
  - **Отступы и типографика:** выверенная шкала: отступы секций (padding/margin) — кратные 4px или 8px (например 1rem, 1.5rem, 2rem, 3rem). Размеры шрифтов — иерархия (например 0.875rem тело, 1rem подзаголовок, 1.25–1.5rem заголовок, 2rem и выше hero). line-height 1.4–1.6 для текста, 1.2 для заголовков.
  - **Мобильный адаптив:** с самого начала: контейнер с `max-width`, при необходимости `padding` по бокам; сетка на малом экране — одна колонка (`grid-template-columns: 1fr` или `repeat(1, 1fr)`), с брейкпоинта (например 640px, 768px) — две-три колонки через media или утилиты (Tailwind: `grid-cols-1 md:grid-cols-2 lg:grid-cols-3`). Кнопки и тач-зоны не менее ~44px по высоте на мобильных.
- **Формы и действия:** форма → `fetch`/`load` к API; обработка ошибок и индикация загрузки (disabled кнопки, сообщение об ошибке).
- **Сборка:** после любых правок `npm run build` должен проходить. Проверять импорты (особенно в Svelte 5 runes), пути к API, отсутствие обращений к браузерному API в SSR-коде.
- **Несколько файлов в одной задаче (компонент + тип + страница):**
  - Общий тип для родителя и дочернего компонента — вынести в `src/lib/types/` (например, `src/lib/types/plan.ts` с `export interface Plan { ... }`) и импортировать в обоих: `import type { Plan } from '$lib/types/plan';`. Не дублировать интерфейс в двух файлах и не вставлять содержимое одного файла в другой.
  - Новый компонент (например, `TariffCard.svelte`) — отдельный файл в `src/lib/components/`. В родителе: `import TariffCard from './TariffCard.svelte';` и использование `<TariffCard {plan} />`. В дочернем — пропсы с типом из `$lib/types/...`.
  - В ответе (diff): **один блок diff = один файл**. Несколько файлов = несколько подряд идущих блоков: первый блок `--- a/src/lib/components/Pricing.svelte +++ b/...`, второй блок `--- /dev/null +++ b/src/lib/components/TariffCard.svelte`, третий при необходимости `--- /dev/null +++ b/src/lib/types/plan.ts`. ЗАПРЕЩЕНО вставлять в тело одного diff текст вида `--- ФАЙЛ: path ---` или содержимое второго файла — это ломает сборку.
  - После добавления компонента/страницы: сразу добавить импорты и ссылку в навигации (`+layout.svelte`), чтобы пользователь видел результат.

### Интеграция: фронт ↔ API ↔ БД

- **Цепочка для новой фичи с БД:**
  1. Схема (Drizzle) — таблицы, типы.
  2. Миграция — SQL в `drizzle/` или `migrations/`, при необходимости сиды.
  3. API — эндпоинты в `src/routes/api/.../+server.ts`, использование `db` и схемы.
  4. Фронт — страница/компонент, вызов API, отображение данных.
  5. Навигация — ссылка в `+layout.svelte` (или в сайдбаре/меню проекта).
- **Не оставлять «мёртвый» код:** если добавлен API — он должен вызываться с фронта или быть явно задокументирован как служебный. Если добавлена страница — на неё должна вести ссылка.
- **Верификация:** после применения патчей автоматически выполняются миграции и сиды (если есть в `package.json`), затем сборка и тесты. Ошибка любого шага возвращается в контекст для повтора.

### Типовые ошибки и как их устранять

- **ERESOLVE / peer dependency conflict** — конфликт версий npm (напр. svelte@4 vs vite-plugin-svelte требующий svelte@5). Решения: (1) `.npmrc` с `legacy-peer-deps=true`; (2) `package.json` → `"overrides": {"svelte": "^4.2.0"}`; (3) downgrade `@sveltejs/vite-plugin-svelte` до ^3.x.
- **Unexpected end of file / Unexpected end of input** — файл обрезан. Вернуть полную версию: все блоки `</script>`, разметку, `</style>` завершить.
- **`<script> must have a closing tag`** — внутри script есть `<` или `</` (напр. SVG в template literal). Вынести в разметку (`{@html}` или отдельный компонент), не хранить HTML/SVG в строке в script.
- **Expected }** (в строке с name:, price: в .svelte) — парсер путает `{` в mustache. Вынести данные в `src/lib/data/plans.ts`, в компоненте только `import { plans }` и разметку. Не `{#each [{...}]}`.
- **Expected ; but found :** — в объектах JS всегда `name: 'X'` с двоеточием, не `|` (pipe). Pipe в Svelte — только для фильтров в шаблоне.
- **Expected ; but found :** — LLM иногда заменяет `:` на `|` в объектах. В JavaScript объектах ВСЕГДА используй `name: 'X'` с двоеточием, не pipe `|`.
- **Expected identifier / Unexpected end of file** — diff обрезан. Вернуть в diff ПОЛНЫЙ файл: закрыть все `}, ], );` перед `</script>`, не применять частичный diff.
- **Expected ; but found :** (name | вместо name:) — в объектах JS используй двоеточие `:` (name: 'X'), не pipe `|`. Pipe в Svelte — только для фильтров в шаблоне. Partial diff/truncated file → вернуть полный diff.
- **Expected ; but found :** (при name/price) — в объектах JS используй двоеточие `:` (name: 'X'), не pipe `|`. Pipe в Svelte — только для фильтров в шаблоне. Частая причина — LLM подменяет `:` на `|`.
- **Expected identifier / Unexpected end of file** — diff обрезан. Вернуть полный diff: закрыть все `}`, `]`, `);` перед `</script>`. Не применять частичный diff.
- **Expected ; but found :** — LLM иногда ошибочно ставит `|` вместо `:` в объектах. В JavaScript объектах всегда `name: 'X'` с двоеточием. Pipe `|` в Svelte — только для фильтров в шаблоне.
- **Expected identifier / Unexpected end of file** — diff обрезан. Вернуть полный diff: закрыть все `}`, `];`, `</script>`, разметку, `</style>`. Не применять частичный diff.
- **Unused CSS selector** — класс определён в `<style>`, но не используется в разметке. Добавить класс в markup или удалить из style. `class:name={cond}` использует класс — если всё же «unused», можно обернуть в `:global(.name)`.
- **Expected ; but found :** — в объектах JavaScript ВСЕГДА `name: 'X'` с двоеточием. Никогда не заменять `:` на `|` (pipe). Pipe в Svelte — только для фильтров в шаблоне. Diff должен быть полным — не обрезать массив plans.
- **Unused CSS selector** — предупреждение, не блокирует сборку. Удалить неиспользуемый селектор или применять класс через `class:name={condition}`.
- **Expected ; but found :** — LLM иногда ошибочно ставит `|` вместо `:` в объектах. В JavaScript объектах всегда `name: 'X'` с двоеточием. Pipe `|` в Svelte — только для фильтров в шаблоне.
- **Expected identifier / Unexpected end of file** — diff обрезан. Вернуть полный diff: закрыть все `}`, `]`, `);` перед `</script>`, не применять частичный diff.
- **Unused CSS selector** — селектор объявлен, но не используется (напр. `class:border-primary-500`). Удалить или использовать в разметке.
- **Expected ; but found :** (при name/price в объекте) — в объектах JavaScript ВСЕГДА используй двоеточие `:` (name: 'X'), НЕ pipe `|`. Pipe в Svelte — только для фильтров в шаблоне.
- **Expected identifier / Unexpected end of file** — diff обрезан. Вернуть полный diff: закрыть все `}, ], );` перед `</script>`, не применять частичный diff.
- **Unused CSS selector** — класс объявлен в `<style>`, но не используется в разметке. Использовать с `class:name={condition}` или удалить.
- **Expected ; but found :** (при name/price в объекте) — LLM иногда подставляет `|` вместо `:`. В объектах JavaScript ВСЕГДА `name: 'X'`, НЕ `name | 'X'`. Pipe `|` в Svelte — только для фильтров в шаблоне.
- **Expected identifier / Unexpected end of file** — diff обрезан. Вернуть полный diff: закрыть все `}`, `]`, `);` перед `</script>`, не применять частичный diff.
- **Unused CSS selector** — класс определён в `<style>`, но не используется. Либо удалить, либо применить (напр. через `class:name={condition}`).
- **Expected ; but found :** / **name \| вместо name:** — в объектах JavaScript ВСЕГДА двоеточие `:` (name: 'X'), не pipe `|`. Pipe в Svelte — только для фильтров в шаблоне. Diff должен содержать полный массив plans, не частичный.
- **Expected identifier / Unexpected end of file** — diff обрезан. Вернуть полный файл: закрыть все `}`, `]`, `];` перед `</script>`, не применять partial diff.
- **Unused CSS selector** (.border-primary-500 и др.) — если класс используется через `class:name={expr}`, можно игнорировать или вынести в :global(.border-primary-500).
- **Expected ; but found :** (при `name |` или `price |`) — LLM иногда подставляет `|` вместо `:` в объектах. В JavaScript объектах ВСЕГДА `name: 'X'`, не `name |`. Pipe `|` в Svelte — только для фильтров в шаблоне.
- **Expected identifier but found end of file** / **Unused CSS selector** — diff обрезан или неполный; лишний CSS. Diff должен быть полным (все `}`, `]`, `);`); удалить неиспользуемые селекторы.
- **Could not resolve / Cannot find module** — нет import или файла. Добавить `import X from './path'` или создать файл `--- /dev/null +++ b/path`.
- **PostgresError: password authentication failed (28P01)** — окружение (DATABASE_URL ≠ credentials в docker-compose). Не исправляется патчами кода; настраивать verify/runner.
- **relation X does not exist** / синтаксическая ошибка в миграции — исправить schema (`src/lib/db/schema.ts`), SQL в `drizzle/` или `migrations/`.
- **Expected ; but found :** (в строке с name/price) — LLM иногда подставляет `|` вместо `:` в объектах. В JavaScript всегда `name: 'X'` с двоеточием, не pipe. Pipe `|` в Svelte — только для фильтров в шаблоне.
- **Partial/truncated diff** — LLM применяет обрезанный diff, файл остаётся невалидным. Diff должен быть полным: закрыть все `}`, `]`, `];` перед `</script>`, включить весь контент.
- **Unused CSS selector** — предупреждение vite-plugin-svelte. Если класс используется в `class:name={condition}` — можно игнорировать или вынести в глобальные стили.

## Архитектура Task Admin

### Мультипроектность

- Один экземпляр Task Admin управляет несколькими проектами.
- Каждый проект — отдельный workspace с собственными TASKS.md, scripts/, .cursor/rules/.
- Runner запускается с `PROJECT_ROOT=<path>` — скрипты выполняются в контексте выбранного проекта.

### Разделение ответственности

- **Backend (Rust)**: API, парсинг TASKS.md, запуск/остановка runner, OpenRouter.
- **Frontend (Svelte)**: UI, выбор проекта, отображение задач, чат.
- **Scripts (Bash)**: Планировщик, исполнитель, ревьюер — привязаны к `PROJECT_ROOT`.

## Проект

### Требования к проекту

1. **TASKS.md** — структура разделов: Пользовательские задачи, Спринт, Баги, Бэклог, Выполнено.
2. **scripts/** — team.sh, task-runner.sh, task-planner.sh, task-reviewer.sh, verify-task.sh.
3. **TASK_REPORTS.md** — генерируется task-runner, формат `## YYYY-MM-DD HH:MM:SS — текст задачи`.
4. **data/admin-project.json** — профиль (опционально): project_path, project_name, auto_research_enabled.

### Скрипты и PROJECT_ROOT

При запуске из Task Admin скрипты получают `PROJECT_ROOT`. В начале team.sh:

```bash
PROJECT_ROOT="${PROJECT_ROOT:-$(cd "$(dirname "$0")/.." && pwd)}"
cd "$PROJECT_ROOT"
```

Это позволяет запускать скрипты как из самого проекта, так и из Task Admin.

## Безопасность

- Не хранить секреты в `data/projects.json`.
- `OPENROUTER_API_KEY` — только в env.
- Пути к проектам — абсолютные, без `..`.

## Производительность

- Rust backend — нативный код, минимальная задержка.
- Svelte 5 — компиляция в vanilla JS, нет VDOM, малый bundle.
- Polling runner status — раз в 5 сек при активном runner.

## Расширение

- Новые API: добавить route в `src/main.rs`.
- Новые экраны: компоненты в `admin-ui/src/`.
- Поддержка других бэкендов (не только OpenRouter): обобщить `tasks::chat_with_openrouter`.

### Чеклист подключения новой фичи

См. `.cursor/rules/task-admin-architecture.mdc` — правило для AI и разработчиков:

1. **Backend**: route в `api_protected` → handler → `Json`/`api_error`
2. **Frontend**: метод в `api.ts` → компонент → подключение в MainView/App (таб, экран, кнопка)
3. **Сразу подключать** — без route нет API, без вызова в api.ts фронт не использует, без ссылки в UI пользователь не видит.
